# GUTSHOT Poker Club
## Business Rules

Версия: MVP 1.0

---

# Общие принципы

- Вся бизнес-логика выполняется только Backend.
- Mini App и Admin Panel не принимают бизнес-решений.
- Все изменения данных происходят через Backend API.
- Registration является центральной сущностью системы.
- Все даты хранятся в UTC.
- Все действия логируются.
- Все изменения происходят внутри транзакций базы данных.

---

# Авторизация

Игрок авторизуется только через Telegram Mini App.

Backend:

- проверяет Telegram Init Data;
- валидирует подпись Telegram;
- ищет пользователя по Telegram ID;
- если пользователь отсутствует — создает User и PlayerProfile;
- создает JWT;
- возвращает токен.

---

# Регистрация на турнир

Игрок может зарегистрироваться только если:

- турнир существует;
- регистрация открыта;
- турнир не начался;
- игрок не заблокирован;
- игрок еще не зарегистрирован.

После успешной регистрации:

- создается Registration;
- статус = REGISTERED;
- создается QRToken;
- отправляется Telegram сообщение.

---

# Лист ожидания

Если свободных мест нет:

создается Registration

status = WAITING

Если место освобождается:

первый игрок из WAITING автоматически получает место.

Ему:

- меняется статус на REGISTERED;
- создается новый QR;
- отправляется Telegram сообщение.

---

# Отмена регистрации

Игрок может отменить регистрацию до начала турнира.

После отмены:

Registration.status = CANCELLED

QRToken становится недействительным.

Если существует WAITING —

место автоматически получает первый игрок.

---

# QR

QR используется только для Check-In.

QR:

- одноразовый;
- криптографически случайный;
- действует 60 секунд;
- после использования становится недействительным;
- автоматически обновляется при открытии экрана "Мой турнир".

Одновременно существует только один активный QR.

---

# Check-In

Администратор сканирует QR.

Backend проверяет:

- QR существует;
- QR не просрочен;
- QR не использован;
- Registration существует;
- Registration.status = REGISTERED.

После успешной проверки:

Registration.status = CHECKED_IN

QR.usedAt = now()

Игрок получает Telegram сообщение.

---

# Начало турнира

Администратор нажимает

"Начать турнир"

↓

Tournament.status = IN_PROGRESS

↓

Все Registration

CHECKED_IN

↓

PLAYING

Игроки со статусами WAITING, CANCELLED и NO_SHOW не участвуют.

---

# Завершение турнира

Администратор вводит итоговые места игроков.

После подтверждения:

для каждой Registration

- записывается place;
- записывается xpAwarded;
- статус становится FINISHED.

После этого:

- создается запись XPHistory;
- обновляется PlayerProfile.xp;
- рассчитывается новый уровень;
- отправляется Telegram сообщение.

После обработки всех игроков:

Tournament.status = FINISHED

Все действия выполняются в одной транзакции.

---

# Уровни

Уровень вычисляется автоматически по XP.

Backend определяет уровень после каждого начисления XP.

Уровень не хранится отдельно.

---

# Рейтинг

Общий рейтинг:

сортировка игроков по общему XP.

Недельный рейтинг:

сумма XP за последние 7 дней.

Отдельные таблицы рейтингов не используются.

---

# Статистика

Статистика рассчитывается автоматически.

Не хранится в базе.

Количество игроков

↓

COUNT(User)

Количество турниров

↓

COUNT(Tournament)

Количество посещений

↓

COUNT(Registration WHERE status = FINISHED)

Победы

↓

COUNT(Registration WHERE place = 1)

---

# Telegram уведомления

Backend автоматически отправляет сообщения:

- успешная регистрация;
- перевод с WAITING в REGISTERED;
- отмена регистрации;
- напоминание за 2 часа до турнира;
- успешный Check-In;
- завершение турнира;
- начисление XP;
- новый уровень.

История сообщений сохраняется в Notification.

---

# Администраторы

Роли:

OWNER

Полный доступ.

ADMIN

Работа с турнирами.

MANAGER

Только регистрация и Check-In.

---

# Ограничения

Игрок не может:

- зарегистрироваться дважды;
- зарегистрироваться после начала турнира;
- использовать QR повторно;
- использовать чужой QR.

Администратор не может:

- изменить результаты после завершения турнира без роли OWNER.

---

# Безопасность

Все запросы проходят JWT авторизацию.

Все административные действия логируются.

QR невозможно подобрать перебором.

Пароли администраторов хранятся только в виде bcrypt-хэша.

Все соединения работают только по HTTPS.

---

# Принципы проекта

Backend — единственный источник истины.

Frontend не содержит бизнес-логики.

Все вычисления выполняются на сервере.

Данные не дублируются.

Статистика вычисляется автоматически.

Каждое действие должно быть воспроизводимым через историю данных.

Проект строится вокруг жизненного цикла Registration.
