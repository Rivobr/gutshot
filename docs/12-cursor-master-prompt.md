# GUTSHOT Poker Club — Cursor Master Prompt

## Общая информация

Ты участвуешь в разработке коммерческого продукта GUTSHOT Poker Club.

Это не учебный проект.

Это Telegram Mini App и CRM-система для реального офлайн-покерного клуба в Санкт-Петербурге.

Цель — создать современный продукт уровня 2026 года с чистой архитектурой, высокой производительностью и качественным UX.

Перед началом любой задачи обязательно изучай документацию проекта в папке `docs/`.

---

# Источники истины (приоритет)

При возникновении противоречий используй следующий порядок приоритетов:

1. Документация (`docs/`)
2. Backend API
3. Prisma Schema
4. Figma Design
5. Figma Export Code

Не придумывай собственные решения, если они противоречат этим источникам.

# Работа с дизайном

В проекте находится полный экспорт интерфейса из Figma.

Путь:

design/figma-export/

Экспорт используется как источник визуального интерфейса.

Перед созданием любого экрана необходимо:

- изучить соответствующий экран в экспорте;
- сохранить внешний вид максимально близко к дизайну;
- сохранить цвета;
- сохранить типографику;
- сохранить размеры;
- сохранить анимации;
- сохранить фирменный стиль клуба.

Разрешается использовать:

- SVG;
- изображения;
- иконки;
- Tailwind-классы;
- анимации;
- стили.

Запрещается копировать архитектуру проекта из экспорта.

---

# Feature-Sliced Design

Использовать следующую структуру:

src/

- app
- shared
- entities
- features
- widgets
- pages
- processes

Не создавать компоненты вне этой структуры.

---

# Компоненты

Каждый компонент должен иметь одну ответственность.

Повторяющиеся элементы необходимо выносить в `shared/ui`.

Большие компоненты необходимо разбивать на небольшие.

Не создавать файлы длиннее ~300 строк без веской причины.

---

# Бизнес-логика

Вся бизнес-логика находится на Backend.

Frontend отвечает только за:

- отображение данных;
- отправку запросов;
- локальное состояние интерфейса;
- анимации.

Не переносить вычисления и правила бизнеса во Frontend.

---

# API

Все запросы проходят через централизованный API Layer.

Не делать запросы напрямую из компонентов.

Использовать TanStack Query для получения и кеширования данных.

---

# Формы

Использовать:

- React Hook Form;
- Zod.

Валидация должна быть как на клиенте, так и на сервере.

---

# Стиль кода

Использовать:

- строгую типизацию;
- понятные имена;
- ранний `return`;
- небольшие функции;
- читаемый код;
- композицию вместо дублирования.

Не использовать `any`, если это можно избежать.

---

# UI

Интерфейс должен полностью соответствовать дизайну GUTSHOT.

Особенности:

- премиальный стиль;
- темная тема;
- золотые акценты;
- плавные анимации;
- минимализм;
- большие карточки;
- качественная типографика.

Все тексты интерфейса должны быть на русском языке, если в документации не указано иное.

---

# Анимации

Использовать Framer Motion.

Анимировать:

- переходы между страницами;
- появление карточек;
- открытие модальных окон;
- изменение XP;
- повышение уровня;
- появление QR;
- загрузку;
- кнопки.

Анимации должны быть плавными и ненавязчивыми.

---

# Производительность

Использовать:

- Lazy Loading;
- Code Splitting;
- Memoization там, где это оправдано;
- оптимизацию повторных рендеров;
- кеширование запросов.

---

# Безопасность

Не хранить секреты в коде.

Использовать JWT.

Все защищенные запросы должны проходить авторизацию.

---

# Документация

Перед началом реализации новой функции:

1. Изучи соответствующие документы в `docs/`.
2. Определи необходимые сущности.
3. Определи API.
4. Определи необходимые компоненты.
5. Только после этого начинай писать код.

# Последовательность работы

При реализации новой функции придерживайся следующего порядка:

1. Изучи документацию.
2. Изучи экспорт Figma.
3. Определи структуру Feature.
4. Создай типы.
5. Создай API.
6. Создай UI-компоненты.
7. Подключи бизнес-логику.
8. Добавь анимации.
9. Проверь адаптивность.
10. Выполни самопроверку.

---

# Правила генерации кода

При каждом ответе:

- сначала кратко объясни план;
- затем предложи структуру файлов;
- затем создай код;
- не изменяй существующую архитектуру без необходимости;
- не удаляй существующий функционал, если это не требуется задачей.

Если информации недостаточно — сначала задай уточняющий вопрос.

# Качество

Каждый созданный файл должен быть готов к использованию.

Не оставляй TODO без явной необходимости.

Не используй заглушки там, где можно реализовать рабочее решение.

# Цель проекта

Создать масштабируемую, поддерживаемую и производительную систему для управления офлайн-покерным клубом GUTSHOT Poker Club, которая будет состоять из Telegram Mini App, Backend API и Admin Panel, с единым премиальным дизайном и чистой архитектурой.
